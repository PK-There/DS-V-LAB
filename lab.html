Stack Data Structure

Definition
A stack is a linear data structure that follows the Last In, First Out (LIFO) principle.
The last element inserted is the first one to be removed.
Insertion and deletion are allowed only from one end, called the top of the stack.

  • Main operations:
    • Push → Insert an element.
    • Pop → Remove the top element.
    • Peek/Top → View the top element without removing it.
  • Basic Concept
    • A stack ensures a strict LIFO order.
    • Elements are added and removed only from the top.
    • Useful for problems requiring reversing order or tracking recent actions.
  • Basic Applications
    • Function Call Management → Call stack.
    • Undo/Redo Operations → In text editors.
    • Expression Evaluation & Syntax Parsing → In compilers.
    • Backtracking Algorithms → Maze or puzzle solving.
  • Implementation
    • Arrays (fixed size).
    • Linked Lists (dynamic size).
  • Standard Operations on Stack
    • Push – Add element to the top. If full (array-based), causes overflow.
    • Pop – Remove top element. If empty, causes underflow.
    • Peek (Top) – Show top element without removing.
    • isEmpty() – Check if stack is empty.
    • isFull() – Check if stack is full (array-based).
    • Size() – Number of elements in stack.
    • Display() – Print all elements from top to bottom.
  • Features of Stack
    • LIFO Structure – Last in, first out.
    • Single-End Operations – Insert/delete only at top.
    • Efficient Memory Use – In linked list implementation.
    • Limited Operations – Only push, pop, peek, etc.
    • Simple to Implement – Easy concept and coding.
    • Static or Dynamic – Arrays (fixed) or Linked Lists (dynamic).
  • Advantages
    • Easy to use and manage.
    • Supports recursion (function calls use system stack).
    • Helps in memory management during recursion.
    • Supports backtracking (undo, maze solving, history).
    • Useful for expression evaluation (e.g., infix → postfix).
  • Disadvantages
    • Limited Access → Only top element accessible.
    • Overflow/Underflow Risks.
    • Fixed Size (arrays) → Must be predefined.
    • Inefficient for Searching → No random access.
    • Memory Issues in Recursion → Deep recursion may cause stack overflow.
  • Algorithm for Stack Operations 
    Start
Initialize empty list: stack = []
Define MAX = 5 (max size).

Push()
If len(stack) == MAX → Overflow.
Else → Insert element at top.

Pop()
If empty → Underflow.
Else → Remove and return top element.

Peek()
If not empty → Show stack[-1].
Else → Print "Empty".

isFull()
If len(stack) == MAX → Print "Full".
Else → Print "Not Full".

isEmpty()
If empty → Print "Empty".
Else → Print "Not Empty".

Display()
If not empty → Show elements (top → bottom).
Else → Print "Empty".

Menu (Repeat Until Exit):
1 → Push
2 → Pop
3 → Peek
4 → isFull
5 → isEmpty
6 → Display
7 → Exit

Infix, Prefix, and Postfix Expressions

Introduction

  • Infix Notation: Operators are written between operands (e.g., A + B).
  • Prefix (Polish) Notation: Operators are written before operands (e.g., + A B).
  • Postfix (Reverse Polish) Notation: Operators are written after operands (e.g., A B +).
👉 Computers prefer prefix or postfix because:
  • No parentheses are required.
  • Operator precedence and associativity are inherently managed.
  • Parsing and evaluation are easier.

Infix to Prefix Conversion

  • Operators come before operands → e.g., + A B.
  • No parentheses required.
  • Uses reverse logic → reverse infix → convert to postfix → reverse result.
  • Easier for compilers to parse and evaluate.
  • Evaluation order → right to left.

Infix to Postfix Conversion

  • Operators come after operands → e.g., A B +.
  • No parentheses required.
  • Evaluated left to right.
  • Uses a stack for managing operators and precedence.
  • Widely used in calculators and interpreters.

Applications

  • Compiler Design: Converts programmer-written infix expressions into prefix or postfix. Simplifies parsing and code generation.
  • Expression Evaluation in Calculators: Scientific calculators and stack-based machines use postfix (RPN). Eliminates parentheses and precedence complexity.

Algorithm for Conversions

Infix → Postfix 
Scan infix expression from left to right.
If symbol is an operand, add to postfix expression.
If symbol is an operator:
    If stack is empty, or top is (, or top has lower precedence, push operator.
    Else, pop higher/equal precedence operators to postfix, then push current operator.
If symbol is (, push it to stack.
If symbol is ), pop and add to postfix until ( is found (discard both parentheses).
Repeat until expression ends.
Pop remaining operators from stack to postfix.
Infix → Prefix 
Reverse the infix expression.
Swap ( with ) and vice versa.
Scan reversed expression from left to right.
If symbol is an operand, add to prefix expression.
If symbol is an operator:
    If stack is empty, or top is (, or operator has higher precedence, push it.
    Else, pop from stack until lower precedence operator is found, then push current operator.
If symbol is (, push it to stack.
If symbol is ), pop to prefix until ( is found (discard both parentheses).
Repeat until expression ends.
Pop remaining operators and add to prefix.
Reverse the result to get final prefix expression.
